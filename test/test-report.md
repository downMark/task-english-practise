- **PRD 依据**：8.3 节——副标题建议为"Click a task to listen and move over words to see Chinese meanings."

### 模块 2：英文段落展示

#### TC-07 段落内容可读性
- **PRD 依据**：7.2 节——段落需按自然阅读形式展示，行高舒适、字号适中
- **测试步骤**：观察主内容区段落展示样式
- **预期结果**：文字清晰，段落宽度合适，不过宽
- **实际结果**：段落在主内容区域居中展示，字号适中，单词间距合理，阅读区域舒适
- **状态**：✅ 通过

#### TC-08 单词粒度拆分渲染
- **PRD 依据**：7.2 节——支持按单词粒度绑定中文释义；10.5 节——将正文按"单词 + 标点"拆分渲染
- **测试步骤**：查看段落中各单词的 DOM 结构
- **预期结果**：单词和标点分开渲染，有词汇的单词包裹为可交互元素
- **实际结果**：段落中有词汇的单词（如 ancient、forest、path、curious、whispers、leaves、courage）渲染为 `<button>` 元素，标点单独渲染为 `<span>`，其余普通词汇为 `<span>`
- **状态**：✅ 通过

### 模块 3：音频播放功能

#### TC-09 点击播放按钮播放音频
- **PRD 依据**：7.3 节——用户点击播放按钮，系统播放该段英文语音
- **测试步骤**：点击"Play"按钮
- **预期结果**：音频开始播放，按钮状态变化
- **实际结果**：点击 Play 后，按钮变为"Pause"，状态文案变为"Now reading aloud"，浏览器 `speechSynthesis.speaking` 为 `true`（使用 Web Speech API TTS）
- **状态**：✅ 通过

#### TC-10 播放状态有明确反馈
- **PRD 依据**：7.3 节——播放中需要有明显状态反馈；9.3 节——音频播放态要明确
- **测试步骤**：点击 Play 后观察页面变化
- **预期结果**：播放按钮文案/图标变为"暂停"，有状态提示
- **实际结果**：按钮从"Play"变为"Pause"，状态文本从"Ready to play"变为"Now reading aloud"，反馈清晰
- **状态**：✅ 通过

#### TC-11 播放结束后可重新播放
- **PRD 依据**：7.3 节——播放结束后恢复为可再次播放状态；"重播"按钮存在
- **测试步骤**：观察"Replay"按钮是否存在
- **预期结果**：存在重播按钮
- **实际结果**：页面始终显示"Play"和"Replay"两个按钮
- **状态**：✅ 通过

#### TC-12 音频实现方式
- **PRD 依据**：7.3 节——优先使用浏览器音频播放能力；若未提供真人录音，可考虑 TTS
- **测试步骤**：检查 DOM 中 `<audio>` 元素及 `speechSynthesis` 状态
- **预期结果**：有可用音频播放机制
- **实际结果**：页面使用浏览器 `Web Speech API (speechSynthesis)` 实现 TTS，DOM 中无 `<audio>` 元素。TTS 方式符合 PRD 备选方案，但不同浏览器音效存在差异（PRD 风险 5 中已提及）
- **状态**：⚠️ 部分通过（使用 TTS 而非真人录音，符合 PRD 备选方案，建议后续替换为预生成音频文件）

#### TC-13 切换任务时旧音频停止
- **PRD 依据**：7.3 节——切换任务时，若上一段音频仍在播放，应自动停止；流程 B
- **测试步骤**：先播放 Task 1 音频，再点击 Task 2
- **预期结果**：旧音频停止，主内容区切换到 Task 2
- **实际结果**：切换到 Task 2 后，`speechSynthesis.speaking` 变为 `false`，TTS 已停止；主内容区更新为"City Lights and Busy Streets"
- **状态**：✅ 通过

### 模块 4：单词悬停释义功能

#### TC-14 有词汇的单词有视觉提示
- **PRD 依据**：7.4 节——对已支持释义的单词可提供轻微视觉提示，例如下划线、点状底纹
- **测试步骤**：观察段落中单词的样式
- **预期结果**：有释义的单词有下划线或其他视觉区分
- **实际结果**：有释义的单词（ancient、forest、path 等）使用 `underline decoration-dotted underline-offset-4` 样式，呈现点状下划线，与普通单词视觉上有区分
- **状态**：✅ 通过

#### TC-15 悬停单词显示 tooltip 中文释义
- **PRD 依据**：7.4 节——鼠标移入即显示中文释义，tooltip 包含英文单词和中文释义
- **测试步骤**：鼠标悬停在"ancient"单词上，检查 tooltip DOM
- **预期结果**：出现包含英文和中文释义的 tooltip
- **实际结果**：DOM 中存在 tooltip 元素，内容为"ancient古老的 · adj."，包含单词原文、中文释义和词性，符合 PRD 要求
- **状态**：✅ 通过

#### TC-16 tooltip 内容完整性
- **PRD 依据**：7.4 节——tooltip 内容至少包含英文单词原文和中文释义；支持词性
- **测试步骤**：查看 tooltip 内容结构
- **预期结果**：包含英文单词 + 中文释义，可选词性
- **实际结果**：tooltip 内容"ancient古老的 · adj."包含单词、中文释义、词性（adj.），内容完整
- **状态**：✅ 通过

#### TC-17 tooltip 不遮挡过多正文
- **PRD 依据**：7.4 节——tooltip 位置不遮挡过多正文
- **测试步骤**：触发 tooltip 后观察位置
- **预期结果**：tooltip 出现在单词附近，不大面积遮挡文章
- **实际结果**：tooltip 使用 `absolute left-1/2 top-full mt-2 w-52` 定位，出现在单词正下方，尺寸合理（w-52 约 208px），不过度遮挡正文
- **状态**：✅ 通过

#### TC-18 移动端点击单词显示释义
- **PRD 依据**：12 节——移动端用点击显示释义（无 hover）
- **测试步骤**：点击"ancient"单词按钮
- **预期结果**：点击后 tooltip 显示
- **实际结果**：单词渲染为 `<button>` 可点击元素，点击后 tooltip 在 DOM 中可见（内容"ancient古老的 · adj."），满足移动端兼容要求
- **状态**：✅ 通过

## 六、总结

英语练习 Web 页面 V1 核心功能已基本实现，产品的三个核心体验——**任务切换**、**语音播放**、**单词悬停释义**——均运行正常，符合 PRD 交互设计要求。页面信息层级清晰，操作路径简洁，用户无需学习成本。

主要遗留问题为任务数量偏少（4 个 vs 建议 5~10 个），建议补充后正式通过 V1 验收。
