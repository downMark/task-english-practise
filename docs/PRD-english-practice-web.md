# 英语练习 Web 页面 PRD

## 1. 文档信息

- 项目名称：英语练习 Web 页面
- 文档类型：产品需求文档（PRD）
- 版本：V1.1
- 日期：2026-06-06
- 目标：设计一个面向英语学习者的轻量级网页，用户可在网页中完成预设英文段落练习，点击段落即可播放语音，鼠标移入单词即可查看中文释义，整体交互保持简洁明了。

## 2. 项目背景

英语初学者或有基础的学习者，在进行段落阅读练习时，常常需要同时完成以下几件事：

- 阅读完整英文段落
- 听标准发音
- 快速查看单词中文释义
- 在尽量少打断阅读节奏的前提下完成练习

传统做法通常需要在多个工具之间切换，例如阅读网页、翻译工具和音频播放器并行使用，学习成本高且注意力容易分散。因此需要一个聚焦单一场景的网页，将“阅读 + 听音 + 悬停释义”整合到一个页面中。

## 3. 产品目标

### 3.1 核心目标

- 提供若干预先配置好的英语练习任务
- 每个任务包含一段英文内容
- 用户点击任务后可播放该段英文语音
- 用户将鼠标移入英文单词时可立即看到中文释义
- 页面操作路径短、理解成本低、视觉表达清晰

### 3.2 产品原则

- 简洁优先：一个页面内完成主要操作，不引入复杂导航
- 学习优先：先让用户快速开始练习，再补充辅助信息
- 打断最少：查词和听音都不应破坏阅读节奏
- 反馈明确：播放状态、悬停释义、当前任务状态都要有清晰提示

## 4. 目标用户

### 4.1 主要用户

- 小学高年级到成人阶段的英语学习者
- 需要通过短文练习提升阅读与听力的用户
- 希望以较低操作成本完成英语输入练习的用户

### 4.2 用户特征

- 英语水平以初级到中级为主
- 对网页操作较熟悉，但不愿进行复杂设置
- 更关注“快速读、快速听、快速查词”

## 5. 使用场景

### 场景 1：短时碎片练习

用户打开页面后，直接看到多个练习任务。点击其中一个任务，展开对应英文段落并开始播放语音，边听边读。

### 场景 2：阅读中即时查词

用户在阅读英文段落时，遇到不认识的单词，将鼠标移到单词上方，即刻出现中文释义提示，无需跳转页面。

### 场景 3：重复跟读

用户听完一段后，希望再次听该段发音，可再次点击播放按钮，进行反复跟读训练。

## 6. 产品范围

### 6.1 本期范围（V1）

- 单页英语练习网页
- 预设固定数量的英文练习任务
- 每个任务有标题、英文段落、音频播放能力
- 支持单词悬停显示中文释义
- 支持基础播放状态反馈
- 适配桌面端，兼顾常规移动端浏览

### 6.2 暂不纳入本期

- 用户登录/注册
- 学习进度云端同步
- 用户自定义上传文章
- 录音跟读评分
- 生词本收藏
- 复杂语法解析
- 多语言切换

## 6.3 技术实现约束

- 前端框架固定为 `Next.js`
- 样式方案固定为 `Tailwind CSS`
- 本期以 Web 页面交付为目标，优先采用成熟、轻量、易部署的实现方式
- 最终运行环境为 AWS EC2 云服务器
- 页面需支持生产环境构建、启动与基础运维

## 7. 核心功能需求

## 7.1 练习任务列表

### 功能说明

页面默认展示预设数量的英语练习任务卡片或列表项，每个任务代表一段英文练习内容。

### 需求点

- 系统预设任务数量，建议初始为 5 到 10 个
- 每个任务需展示：
  - 任务编号
  - 任务标题
  - 难度标签（如初级/中级，可选）
  - 简短描述或段落预览（可选）
- 默认高亮第一个任务，降低用户启动成本

### 交互要求

- 点击任务项可切换当前练习内容
- 当前任务应有明确选中态
- 切换任务后，正文区域同步更新

## 7.2 英文段落展示

### 功能说明

页面主区域展示当前选中的英文段落内容，作为阅读和听音的核心区域。

### 需求点

- 段落按自然阅读形式展示
- 单词之间需保留合理间距
- 支持按单词粒度绑定中文释义
- 正文区域应具备较好的可读性，例如：
  - 行高舒适
  - 字号适中
  - 段落宽度不过长

### 交互要求

- 当前段落加载后立即可阅读
- 不要求用户额外点击进入阅读模式

## 7.3 点击播放语音

### 功能说明

用户点击当前任务的播放按钮，或点击任务本身指定区域后，系统播放该段英文语音。

### 需求点

- 每个任务对应一段完整语音
- 支持播放、暂停、重新播放
- 同一时间仅允许一段音频播放
- 切换任务时，若上一段音频仍在播放，应自动停止

### 交互要求

- 播放按钮文案或图标清晰，例如“播放”“暂停”
- 播放中需要有明显状态反馈
- 播放结束后恢复为可再次播放状态

### 建议实现方式

- 优先使用浏览器音频播放能力
- 若未提供真人录音，可在技术实现阶段考虑文本转语音（TTS）

## 7.4 单词悬停显示中文释义

### 功能说明

用户将鼠标移入正文中的某个英文单词时，显示该单词对应的中文释义提示。

### 需求点

- 释义以 tooltip 浮层形式展示
- tooltip 内容至少包含：
  - 英文单词原文
  - 中文释义
- 常见标点需要与单词分离处理，避免影响 hover 精度
- 一个单词可支持多个中文义项，但 V1 建议只展示 1 到 2 个核心义项，避免信息过载

### 交互要求

- 鼠标移入即显示，移出即消失
- tooltip 位置不遮挡过多正文
- 同一时刻只展示一个 tooltip
- tooltip 出现与消失要足够迅速，不造成拖沓感

### 易用性要求

- 对已支持释义的单词可提供轻微视觉提示，例如下划线、点状底纹或文字颜色变化
- 提示样式要克制，不能影响整段阅读

## 8. 页面结构

### 8.1 推荐页面布局

- 顶部区域：页面标题 + 简短说明
- 左侧或上方区域：练习任务列表
- 主内容区域：当前英文段落
- 主内容辅助区域：播放按钮、当前任务信息

### 8.2 信息层级

- 第一优先级：当前练习内容
- 第二优先级：播放操作
- 第三优先级：任务切换
- 第四优先级：提示说明

### 8.3 推荐首屏内容

- 页面标题，例如“English Practice”
- 一句简短引导语，例如“点击任务开始朗读练习，移入单词查看中文释义”
- 默认选中的第一个任务
- 当前任务正文与播放控件

## 9. 交互设计要求

## 9.1 简洁原则

- 页面层级不超过 2 层
- 不使用复杂弹窗流程
- 不设置冗余筛选条件
- 所有核心操作尽量在首屏完成

## 9.2 关键交互流程

### 流程 A：开始练习

1. 用户进入页面
2. 看见任务列表和默认任务内容
3. 点击播放按钮
4. 听音并阅读
5. 鼠标移入陌生单词查看中文释义

### 流程 B：切换任务

1. 用户点击另一个任务
2. 页面更新为新任务的英文段落
3. 若旧音频正在播放则自动停止
4. 用户点击新任务播放按钮继续练习

## 9.3 状态反馈

- 任务选中态：明确
- 音频播放态：明确
- hover 释义态：即时
- 加载失败态：简单提示，例如“音频加载失败，请稍后重试”

## 10. 内容设计要求

## 10.1 任务内容结构

每个任务建议包含以下字段：

- `id`：任务唯一标识
- `title`：任务标题
- `level`：难度级别
- `text`：英文段落原文
- `audioUrl`：音频地址或 TTS 配置来源
- `words`：单词释义映射表

## 10.2 单词释义数据结构建议

可按如下思路组织：

- 单词原文
- 中文释义
- 可选词性
- 可选发音信息

示例：

```json
{
  "word": "journey",
  "meaning": "旅行；旅程",
  "partOfSpeech": "noun"
}
```

## 10.3 文本内容要求

- 段落长度建议控制在 50 到 150 个英文单词之间
- 句式尽量自然，适合朗读
- 难度需分层，便于用户循序渐进
- 单词释义覆盖率建议达到正文核心词汇的 80% 以上

## 10.4 推荐数据组织方式

为便于 `Next.js` 项目维护，建议将练习任务内容配置化存储。

推荐方式：

- 将任务数据放在本地 JSON 或 TypeScript 配置文件中
- 页面渲染时直接读取本地数据源
- 单词释义与正文内容保持同一任务维度管理
- 音频地址支持本地静态资源路径或远程资源地址

建议目录思路：

- `app/`：页面路由与页面结构
- `components/`：任务列表、音频播放器、单词 tooltip 等组件
- `data/`：练习任务与释义数据
- `public/audio/`：静态音频资源
- `lib/`：分词、文本处理、播放控制等工具函数

## 10.5 技术实现建议

### Next.js 建议

- 优先使用 `App Router`
- 练习页面可优先采用静态内容渲染，降低实现复杂度
- 预设任务内容建议在构建阶段即可读取，无需额外后端接口

### Tailwind CSS 建议

- 使用 Tailwind CSS 完成页面布局、选中态、按钮态、tooltip 样式
- 通过统一设计 token 保持页面简洁一致
- 响应式断点至少覆盖桌面端与常见手机宽度

### 音频实现建议

- 优先使用 HTML5 `audio` 能力
- 若采用浏览器 TTS，需要评估不同浏览器兼容性
- 若对一致性要求更高，建议使用预生成音频文件

### 单词释义实现建议

- 将正文按“单词 + 标点”拆分渲染
- 对存在释义的单词包裹 hover/点击交互
- tooltip 组件建议支持桌面 hover 与移动点击双模式

## 11. 非功能需求

## 11.1 性能要求

- 页面首屏加载时间尽量控制在 2 秒内
- 任务切换响应时间尽量低于 300ms
- hover 释义展示延迟应尽量低于 100ms
- 播放点击后应尽快响应，避免超过 500ms 的明显等待

## 11.2 兼容性要求

- 支持主流现代浏览器
- 优先适配 Chrome、Edge、Safari
- 在桌面端有完整体验
- 在移动端至少保证内容可浏览、可播放；若移动端无 hover，可改为点击单词显示释义

## 11.3 可维护性要求

- 任务内容和释义数据应与页面结构解耦
- 便于后续新增任务
- 音频资源和文本内容支持独立维护

## 11.4 工程与部署要求

- 项目代码需支持通过 `Next.js` 标准命令完成开发、构建与启动
- 样式编译依赖 `Tailwind CSS`，生产环境构建后页面样式必须完整可用
- 生产环境部署目标为 Linux 系统上的 AWS EC2 实例
- 应提供清晰的环境变量管理方式，如 `.env.production`
- 应支持通过反向代理方式对外提供访问能力
- 部署方案需可重复执行，避免依赖手工修改大量服务器配置

## 12. 移动端补充要求

由于移动端不存在标准鼠标悬停行为，需提供替代方案：

- 单词点击显示中文释义
- 点击空白区域关闭释义
- 点击其他单词时切换释义内容

该要求不改变桌面端“鼠标移入显示释义”的核心设计，只是为移动端做兼容补充。

## 13. UI 设计建议

### 13.1 视觉方向

- 风格轻量、干净、偏学习工具属性
- 避免信息拥挤
- 重点突出当前段落与播放操作

### 13.2 颜色建议

- 主色用于高亮当前任务和播放按钮
- 中性色用于正文和说明文案
- tooltip 使用对比清晰但不刺眼的底色

### 13.3 字体与排版建议

- 英文正文需使用清晰、适合阅读的字体
- 中文释义字号略小于正文，但保证可读
- 段落宽度建议控制在舒适阅读范围内，避免过宽

## 13.4 基于 Tailwind CSS 的界面要求

- 页面风格保持轻量，但需要有明确的视觉层次
- 任务列表选中态、按钮悬停态、tooltip 状态要有一致的 Tailwind 样式规范
- 不追求复杂动画，优先保证交互反馈直接、快速
- 页面组件尽量模块化，便于后续在 `Next.js` 中拆分复用

## 14. 技术方案概述

### 14.1 推荐架构

- 前端应用：`Next.js`
- UI 样式：`Tailwind CSS`
- 数据来源：项目本地静态任务配置
- 音频来源：本地静态音频或预设远程音频
- 部署环境：AWS EC2
- 反向代理：Nginx（推荐）

### 14.2 推荐部署形态

推荐采用以下部署方式：

1. 在本地完成 `Next.js` 项目开发
2. 执行生产构建
3. 将项目部署至 AWS EC2
4. 在 EC2 上通过 Node.js 运行 `Next.js` 生产服务
5. 使用 Nginx 反向代理到 `Next.js` 服务端口
6. 绑定域名与可选 HTTPS

### 14.3 EC2 环境建议

- 操作系统建议使用 Ubuntu LTS
- 安装 Node.js LTS 版本
- 安装进程守护工具，如 PM2（推荐）
- 安装 Nginx 作为反向代理
- 配置安全组，仅开放必要端口，如 22、80、443

### 14.4 部署交付要求

- 代码仓库可在新机器上完成安装与启动
- 部署文档应包含环境准备、启动命令、重启命令、日志查看方式
- 若使用 PM2，应提供统一启动配置
- 若使用域名，应说明 DNS 与 Nginx 配置步骤

## 16. 验收标准

满足以下条件可视为 V1 验收通过：

1. 页面能展示预设数量的英语任务
2. 点击任一任务后，能正确显示其英文段落
3. 点击播放按钮后，能播放该段英文语音
4. 播放中切换任务时，旧音频会停止
5. 鼠标移入已配置的英文单词时，能显示中文释义
6. tooltip 展示稳定，不明显遮挡阅读内容
7. 页面整体交互无需学习成本，用户首次进入即可理解如何使用
8. 桌面端体验完整，移动端至少可通过点击单词查看释义
9. 项目基于 `Next.js + Tailwind CSS` 实现
10. 项目可在 AWS EC2 服务器上成功部署并访问
11. 服务重启后页面仍可正常提供访问
12. 静态任务数据、音频资源和释义数据可独立维护

## 17. 风险与注意事项

### 风险 1：单词切分复杂

英文段落中存在大小写、缩写、标点连用等情况，可能导致单词 hover 识别不准确。

### 应对建议

- 在前端渲染阶段做统一分词处理
- 将标点与单词拆分渲染

### 风险 2：音频资源准备成本

若每段都使用独立录音，前期制作成本较高。

### 应对建议

- V1 可优先接入 TTS
- 后续再替换为人工录音

### 风险 3：移动端缺少 hover

手机端无法完全复现桌面体验。

### 应对建议

- 使用点击单词显示释义作为兼容方案

### 风险 4：EC2 环境配置复杂

服务器环境、Node.js 版本、Nginx 配置和进程守护方式不统一，可能导致部署失败或重启后服务不可用。

### 应对建议

- 固定 Node.js LTS 版本
- 采用 PM2 管理进程
- 将部署步骤文档化
- 在上线前完成一次从零部署演练

### 风险 5：TTS 与浏览器兼容性差异

若直接依赖浏览器语音能力，不同浏览器与设备可能出现音色、延迟或可用性差异。

### 应对建议

- V1 优先评估预生成音频资源
- 若使用 TTS，需预先验证 Chrome、Safari、Edge 的表现

## 19. 结论

本产品 V1 应聚焦一个非常明确的目标：让用户用最低操作成本完成“读一段英文、听一段英文、查单词中文释义”的基础练习。页面设计应坚持轻量、直观、低打扰原则，避免不必要功能堆叠。只要任务切换、语音播放和悬停释义三个核心环节体验顺畅，这个网页就具备清晰的产品价值与较强的可用性。
